## 3. 1分钟讲解稿

这个项目来源于我的无线通信科研课题。原来的GRPO代码需要手工修改Python参数运行，不方便管理和复现实验，所以我把它改造成Java与Python协作的异步仿真平台。

Spring Boot后端负责认证、场景、任务状态和结果。提交任务时保存不可变场景快照，并用幂等键和唯一约束防止重复创建；任务与Outbox事件在同一个MySQL事务中写入，再通过RabbitMQ分发。Python Worker加载预训练GRPO权重完成3设备RSMA吞吐量推理，通过内部接口回传结果，Java提交业务结果后才ACK消息。

项目还实现了消息重试与死信、Redis缓存和Lua限流。目前97项Java测试、10项Python测试全部通过，并完成了真实基础设施和CUDA推理的全链路验证。

## 4. 3分钟讲解稿

### 4.1 项目背景

这个项目来源于我的研究生科研课题，场景是绿色能量与无线射频混合供能网络。我已经有能够运行的GRPO训练、评估代码和预训练权重，但最初只能修改Python脚本里的参数后手工运行。它缺少用户和场景管理，也无法可靠地提交任务、观察执行状态、处理失败或者追踪一次结果到底使用了哪份权重和随机种子。

因此，我做了一个无线通信仿真实验管理与异步任务调度平台。目标不是把科研模型包装成一个简单接口，而是建立一条可追踪、可重试、可复现的工程链路。

### 4.2 总体架构和主流程

平台采用模块化单体Java后端加独立Python Worker。Java负责业务规则和数据一致性，Python只负责模型计算，MySQL是最终事实来源，RabbitMQ负责异步传递，Redis承担缓存和入口限流。

用户登录后可以创建无线通信场景。提交任务时，系统会保存一份不可变场景快照，避免后续修改场景影响历史实验，同时使用`Idempotency-Key`、请求摘要和数据库唯一约束吸收重复提交。

任务和待发送的Outbox事件会在同一个MySQL事务中写入，解决数据库已经提交、消息却没有成功发送的问题。后台发布器领取Outbox事件后把消息发送到RabbitMQ，并结合Publisher Confirm、Return、租约恢复和指数退避处理发送失败。

Python Worker使用手动ACK消费任务。它先通过内部接口向Java领取执行轮次和场景快照，完成算法、目标、设备数量及单位校验，再加载预训练GRPO权重进行确定性评估。推理结果回调Java后，Java在同一个事务中更新任务状态、执行记录并插入唯一结果；只有Java确认业务数据已经提交，Python才ACK RabbitMQ消息。

### 4.3 两个主要技术难点

第一个难点是可靠消息和重复执行。RabbitMQ只能提供至少一次投递，消息、HTTP回调和MySQL也不在同一个分布式事务里，所以项目没有宣称Exactly Once。我允许消息重新投递，再通过任务状态、`attemptNo`、条件更新以及执行记录和结果表的唯一约束，保证重复消息不会产生第二份业务结果。准确口径是“At-Least-Once Delivery加Effectively-Once Business Effect”。

第二个难点是Java与Python之间的职责和数据契约。如果Python直接修改业务数据库，两个进程会共同维护任务状态和事务，容易出现不一致。因此Python只负责推理，通过`claim`、`complete`和`fail`三个受Worker Token保护的内部接口与Java交互。场景中的W、mW、mJ和Mb也会显式映射，结果记录模型ID、权重SHA-256和seed，确保能够追踪一次结果的来源。

### 4.4 其他工程设计

在查询侧，我用Redis Cache Aside缓存任务详情，状态更新事务提交后删除缓存；Redis不可用或缓存损坏时回源MySQL。提交入口使用Lua脚本完成用户维度的原子限流，Redis故障时采用Fail Open，因为最终仍有数据库幂等规则保护业务正确性。

项目还保留了Java Mock Worker，在没有CUDA和模型权重的环境下也能验证完整后端链路，但结果会明确标记为`JAVA_MOCK`和非科研结果，避免和真实GRPO结果混淆。

### 4.5 验收结果和项目边界

目前Java侧97项测试、Python侧10项测试全部通过，覆盖认证、场景、任务、幂等、Outbox、RabbitMQ、Redis以及Worker回调。我也使用真实MySQL、RabbitMQ、Redis、Conda环境和CUDA完成了完整链路验证，相同权重、场景和seed可以复现相同的吞吐量指标。

当前版本只加载预训练权重，不提供在线训练；真实模型只支持3设备RSMA吞吐量评估，AoI保持为空；项目也没有前端、对象存储和压测结论。这些属于当前版本明确保留的范围边界。

## 6. 面试官打断时的最短回答

### 为什么不用Python直接做完整后端？

科研代码使用Python和PyTorch最合适，但任务状态、权限、幂等、消息可靠性和结果事务更适合作为统一业务边界管理。我让Java负责业务事实，Python专注计算，避免两个进程共同修改数据库。

### 为什么引入RabbitMQ？

模型推理不应该占用HTTP请求线程。RabbitMQ把任务受理和执行解耦，也提供缓冲、手动ACK、重试和死信能力。但消息队列本身不能保证业务只执行一次，所以还必须配合数据库幂等。

### 为什么有RabbitMQ还需要Outbox？

数据库事务和RabbitMQ发布不是同一个事务。如果先写数据库再发消息，进程可能在两步之间崩溃。Outbox让任务和事件先在同一个数据库事务中提交，再异步发送并重试，从而不丢失已受理任务。

### 项目如何保证幂等？

入口用幂等键、请求摘要和唯一索引吸收重复HTTP提交；消费侧用任务状态、执行尝试号、条件更新和唯一约束吸收重复消息和重复回调。项目保证业务结果不会重复，不宣称消息只投递一次。

### GRPO是你在平台里训练的吗？

不是。当前平台加载我已有的预训练权重进行确定性评估，没有实现在线训练。训练代码仍作为科研代码保留，但不属于当前工程闭环。
