# 阶段9：预训练GRPO模型推理设计

## 目标与边界

阶段9不在线训练模型，只把已经训练完成的GRPO权重接入异步任务平台。第一版范围固定为：

- 3个设备、RSMA接入、吞吐量目标；
- Python加载可信本地权重，创建环境并完成确定性评估；
- 只返回吞吐量，AoI在JSON和MySQL中均为`null`；
- `P_wpt`为W，`P_max`为mW，能量和电池为mJ，数据为Mb；
- 天线数保留为元数据，当前模型不使用；
- 不改变原训练脚本和`RsmaEnv.DEFAULTS`中的默认物理参数。

## 五个实现步骤

1. 9.1：整理无硬编码、可传场景、输出标准JSON且结果可复现的独立推理函数；
2. 9.2：固定RabbitMQ消息、Java领取响应、成功/失败回调及单位映射；
3. 9.3：实现Worker Token、领取/成功/失败三个Java内部接口；
4. 9.4：实现Python RabbitMQ消费者，调用GRPO推理并回调Java；
5. 9.5：验证重复领取、重复完成、模型边界、单位映射、失败分类和全量回归。

## 为什么删掉进度、心跳和运行中取消接口

一轮评估通常只需数秒。为每回合发送进度会增加HTTP请求、事务和故障状态，却没有明显用户价值。Python链路只在领取时写一次`started_at/heartbeat_at`，结束时写成功或失败；若进程崩溃，阶段7已有的超时恢复器负责兜底。任务开始前的取消仍由Java状态机吸收，但第一版不承诺在几秒推理中途抢占Python计算。

## 数据职责

Python不连接MySQL。RabbitMQ只分发轻量任务引用；Java读取不可变场景快照并维护任务、执行记录、结果事务和缓存失效；Python只负责读取输入、加载权重、推理和回调。这样不会出现Java与Python同时修改任务状态的双写问题。

## 准确项目口径

可以表述为“通过RabbitMQ和内部HTTP API接入Python/PyTorch预训练GRPO模型，完成3设备RSMA场景的异步吞吐量评估”。不能表述为“平台在线训练GRPO”“支持任意设备数量”或“实现AoI评估”。
